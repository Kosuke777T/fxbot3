# 改善点3: トレーリングストップ TP連動修正

## 問題
`trailing_tp_enabled=true` 時、TP側トレーリングが実質的に動作しなかった。

### 原因分析
旧実装では、価格がTPを超えた時点でMT5の`tp=0`（解除）を送信する方式だった。
しかしMT5ではTP=0にしても「TPなし」扱いとなるだけで、利益を延伸するトレーリング効果がなかった。
SLトレーリングとTP側ロジックが独立しており、SLが移動してもTPは元の位置のまま固定されていた。

### 影響
- TPに到達した瞬間にTP=0で決済保護がなくなる
- SLが引き上がってもTPとの距離が縮まり、利益延伸効果がない

## 修正内容

### workers.py `_update_trailing_stops()`
- **削除**: TP到達時にTP=0にする独立ロジック（旧827-842行）
- **追加**: SLトレーリング発動時にTPも連動移動
  - SL移動量（`sl_shift = new_sl - old_sl`）を計算
  - TPにも同じ移動量を加算: `new_tp = old_tp + sl_shift`
  - `modify_position(ticket, sl=new_sl, tp=new_tp)` で一括送信
  - `trailing_tp_enabled=false` 時はTP移動なし（従来通りSLのみ）

### 動作イメージ（BUY例）
```
エントリー: 150.000, SL: 149.500, TP: 151.000 (SL-TP距離=1.500)
↓ 価格上昇、SLトレーリング発動
SL: 149.500 → 150.200 (+0.700)
TP: 151.000 → 151.700 (+0.700)  ← TP連動
↓ さらに上昇
SL: 150.200 → 150.800 (+0.600)
TP: 151.700 → 152.300 (+0.600)  ← TP連動
```

## バックテストエンジンとの整合
バックテストエンジン(`backtest/engine.py`)はTP到達後にSLをロックアップする独自方式を使用。
ライブ取引(workers.py)ではMT5の制約上、SL-TP連動移動方式を採用。
