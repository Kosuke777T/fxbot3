# 改善点3: トレーリングストップ TP連動修正

## 問題
`trailing_tp_enabled=true` 時、TP側トレーリングが実質的に動作しなかった。

### 原因分析
旧実装では、価格がTPを超えた時点でMT5の`tp=0`（解除）を送信する方式だった。
しかしMT5ではTP=0にしても「TPなし」扱いとなるだけで、利益を延伸するトレーリング効果がなかった。
SLトレーリングとTP側ロジックが独立しており、SLが移動してもTPは元の位置のまま固定されていた。

### 影響
- TPに到達した瞬間にTP=0で決済保護がなくなる
- SLが引き上がってもTPとの距離が縮まり、利益延伸効果がない

## 修正内容

### workers.py `_update_trailing_stops()`
- **削除**: TP到達時にTP=0にする独立ロジック（旧827-842行）
- **追加**: SLトレーリング発動時にTPも連動移動
  - SL移動量（`sl_shift = new_sl - old_sl`）を計算
  - TPにも同じ移動量を加算: `new_tp = old_tp + sl_shift`
  - `modify_position(ticket, sl=new_sl, tp=new_tp)` で一括送信
  - `trailing_tp_enabled=false` 時はTP移動なし（従来通りSLのみ）

### 動作イメージ（BUY例）
```
エントリー: 150.000, SL: 149.500, TP: 151.000 (SL-TP距離=1.500)
↓ 価格上昇、SLトレーリング発動
SL: 149.500 → 150.200 (+0.700)
TP: 151.000 → 151.700 (+0.700)  ← TP連動
↓ さらに上昇
SL: 150.200 → 150.800 (+0.600)
TP: 151.700 → 152.300 (+0.600)  ← TP連動
```

## バックテストエンジンとの整合
バックテストエンジン(`backtest/engine.py`)はTP到達後にSLをロックアップする独自方式を使用。
ライブ取引(workers.py)ではMT5の制約上、SL-TP連動移動方式を採用。

## 改善点

### 1. SL損失圧縮（最優先）

SL平均 -978円 vs TP平均 +1,151円。SL:TP比が約0.85:1で、理想的な1:2以上に遠い。

- **対策A**: SL幅をATR倍率で動的に縮小（現在の固定倍率を見直し）
- **対策B**: 信頼度が低い（<0.40）時のエントリーを抑制
- **対策C**: トレーリングストップの発動距離を短縮し、早期にSLを引き上げ

### 2. EURJPY効率改善

8件中4件SL。平均PnL +106円と他通貨ペア比で効率が低い。

- TP: +350, +830, +830, +1,620（平均+908）
- SL: -330, -690, -1,080, -650（平均-688）
- 勝率50%で利益が出ているが、SL幅のばらつきが大きい

### 3. 予測値帯域の活用改善

全17件の予測値（prediction）が -0.00037 〜 +0.00036 の狭い範囲。信頼度（confidence）は0.34〜0.62。

- 信頼度0.4未満のトレード: 5件中3件がSL → フィルター閾値を0.40に引き上げ検討
- 予測値の絶対値が小さいトレードほどSL率が高い傾向

### 4. MT5接続・運用エラー削減

ログから確認されたエラーパターン:

| エラー | 原因 | 件数 |
|--------|------|------|
| retcode=10025 "No changes" | SL/TP変更値が既存値と同一 | 多数 |
| retcode=10018 "Market closed" | 市場閉鎖中にSL/TP変更を試行 | 20件以上（3/5 07時台に集中） |
| No IPC connection | MT5ターミナル未起動/接続切れ | 複数日で発生 |
| SSL証明書エラー | Slack通知のSSL検証失敗 | 2件 |

- **対策**: 市場閉鎖判定を追加し、閉鎖中はSL/TP変更をスキップ
- **対策**: SL/TP変更前に現在値と比較し、変更不要なら送信しない
- **対策**: IPC接続断時の自動再接続リトライ

### 5. シンボル名不整合修正

初期設定で `USDJPY`（サフィックスなし）を指定した際に「シンボル情報取得失敗」が発生。
実際のブローカーシンボルは `USDJPY-`（ハイフン付き）。

- 設定UIでサフィックス自動補完 or バリデーションを追加

### 6. WFO評価期間不足への対応

ログに「データ期間(49日)がWFOウィンドウ(180+30日)より短いため自動調整」の警告あり。
初回WFOは0フォールド0トレードで実質的に未評価。

- 自動調整ロジックは実装済み（train=34日, test=15日に縮小）
- 運用日数が増えれば自然に解消するが、短期間でもWFO結果の信頼度表示があると良い
