## 2. TP/SL 乗数の見直し ⚡高優先

### 現状
- SL = ATR × 2.0
- TP = ATR × 3.0（リスクリワード 1:1.5）

### 問題点

**TP到達率が低すぎる（21.6%）**

TP が 3.0×ATR と遠いため、相場がTPに届く前に反転しSLが発動するケースが多い。
理論上 R:R=1.5 で損益分岐点は 40%（1/(1+1.5)=40%）だが、
実際の勝率 43.2% は分岐点ギリギリであり、余裕がない。

**SL損失が大きすぎる**

SL発動時の平均損益 -759 は全体の収益を圧迫している（15件×-759 = -11,381）。

### 改善案

**案A: TP を小さくして到達率を上げる**
```yaml
# atr_tp_multiplier: 3.0 → 2.0
risk:
  atr_sl_multiplier: 2.0
  atr_tp_multiplier: 2.0  # R:R=1:1 に変更（損益分岐点50%）
```
TPが近くなり到達率が上がる見込み。勝率が50%を超えれば収益に貢献。

**案B: SL を広げてノイズで刈られにくくする**
```yaml
risk:
  atr_sl_multiplier: 2.5  # 2.0→2.5 に拡大
  atr_tp_multiplier: 3.5  # 比率維持
```
SLが広がると損失時は大きくなるが、ノイズでの損切りが減る可能性。

**推奨**: 短期的には **案A（TP=2.0）** を試す。
TP近接で勝率が50%を超えるかバックテストで確認。

## 3. トレーリングストップ設定の見直し ⚡高優先

### 現状
```yaml
risk:
  trailing_activation_atr: 1.0  # 利益が1×ATRになったらトレーリング開始
  trailing_atr_multiplier: 1.5  # 高値から1.5×ATR後方にSLを配置
```

### 問題点：ブレークイーブン未達のトレーリング

トレーリング発動時（利益=1×ATR）、SLは「高値 - 1.5×ATR」に設定される。

```
例: entry=150.000, ATR=0.050
発動時高値: 150.050（entry+1ATR）
トレーリングSL: 150.050 - 0.075（1.5ATR）= 149.975
→ エントリー価格（150.000）より下！
```

**つまりトレーリングが発動しても、価格が高値から反転すると損失で終わる。**

13件のTrailing決済のうち平均+73と小幅な利益にとどまっているのはこのため。

### 改善案

**発動距離 ≥ トレーリング距離** に設定すること（ブレークイーブン保証）。

```yaml
risk:
  trailing_activation_atr: 2.0  # 2×ATR利益で発動（現行1.0→2.0）
  trailing_atr_multiplier: 1.0   # 高値から1×ATR後方（現行1.5→1.0）
```

発動後のワーストケース:
```
高値: entry + 2×ATR
トレーリングSL: (entry+2ATR) - 1ATR = entry + 1ATR（+1ATR利確保証）
```

これにより「せっかく利益が出てもトレーリングで損失」が解消される。

## 4. 通貨ペア別対応 ⚡高優先

### EURJPY の問題

- 勝率 25%（12取引中3勝のみ）
- 合計損益 -1,080
- EURJPY はスプレッドが広く、EUR とJPY 両方の要因が絡む複合ペア
- M5スケールのノイズが大きい可能性

**改善案**:
1. **EURJPY のシグナル閾値を引き上げる**: `min_prediction_threshold` をペア別に設定、EURJPYは0.0005以上に
2. **ADXフィルターを強化**: EURJPYは`min_adx: 30`（全体の25より高め）
3. **スプレッドフィルター強化**: EURJPYは`max_spread_pips: 2.0`（全体の3.0より厳しく）
4. **短期的にEURJPYを無効化**し、モデル再学習後に様子を見る

### USDJPY の問題

- 勝率 42.9%（14取引中6勝）だが合計損益 -1,771
- 勝ちは小さく、負けが大きいパターン（SELL偏重：14取引中11がSELL）
- 2026年2〜3月の相場が円安方向だったならSELL USDJPYは不利

**改善案**:
1. **高位足トレンドフィルター追加**: H4/D1のトレンド方向と逆行するシグナルを除外
2. **SELL USDJPY を制限**: 上位足がUSD強気（上昇トレンド）のときSELLシグナルを無効化

## 5. BUY/SELL バイアスの解消 🔸中優先

### 現状

全37取引のうちSELL=68%（25件）、BUY=32%（12件）。

期間中の相場がUSD下落・EUR上昇トレンドだったならば、
- SELL USDJPY → 逆張り（不利）
- SELL EURUSD → 順張り（有利）
- BUY EURJPY, BUY USDJPY → 順張り（有利）

の構造になっており、SELLの多さ自体よりも**上位足トレンドとの整合性**が問題。

### 改善案：上位足トレンドフィルターの追加

`strategy/signal.py` の `generate_signal()` に、H4またはD1のEMAトレンド方向フィルターを追加する。

```python
# H4のEMA50が上向き → BUYのみ許可
# H4のEMA50が下向き → SELLのみ許可
# H4がレンジ（ADX<20）→ 両方許可
```

既存の特徴量に `h4_ema_50` は含まれているため実装コストは低い。

## 6. LightGBM 学習の改善

### 6-1. ラベルの不整合 ⚡高優先

**現状の問題**:

`model/labeling.py` のTriple Barrier パラメータ:
```python
sl_mult: float = 2.0  # TP/SL 対称
tp_mult: float = 2.0
```

実際の取引設定 (`config.yaml`):
```yaml
atr_sl_multiplier: 2.0
atr_tp_multiplier: 3.0  # ← 非対称！
```

**学習時はSL=TP=2×ATR（対称）で正解ラベルを作りながら、**
**実際はTP=3×ATR（非対称）で取引している。**

これはモデルが学習した「TP到達」の定義と実際の取引の「TP到達」の定義が異なることを意味する。

**改善案**: `labeling.py` のデフォルト値を実際の設定に合わせる。
```python
# labeling.py
sl_mult: float = 2.0
tp_mult: float = 3.0  # ← 実際の設定と一致させる
```

または `config.yaml` の `atr_sl_multiplier`, `atr_tp_multiplier` をラベリングに渡すようにする。

### 6-2. Regression → Classification への切り替え検討 🔸中優先

**現状**: `mode: regression`（全取引の confidence=1.00 が証拠）

Regressionモードでは:
- 予測値 = 未来の対数収益率（0.0002など小さな値）
- 信頼度 = 常に1.0（意味なし）
- `min_confidence: 0.0` が効かない

Classificationモード（Triple Barrier）に切り替えると:
- 予測 = UP/DOWN/NEUTRAL の3クラス確率
- 信頼度 = max_prob（0〜1の実数）が使えるようになる
- `min_confidence: 0.55` 程度で低確信度トレードを除外できる

**バックテスト比較を推奨**:
```yaml
model:
  mode: classification  # regression → classification

trading:
  min_confidence: 0.55  # 低信頼度除外（現行0.0）
```

### 6-3. クラス不均衡対策 🔸中優先

Triple Barrier ラベリングでNEUTRAL(0)が多数を占める場合、
そのまま学習するとモデルがNEUTRALに偏る。

**確認方法**:
```python
y.value_counts(normalize=True)
# 理想: UP≈30%, DOWN≈30%, NEUTRAL≈40%
# 問題: NEUTRAL>60%
```

**改善案**:
```python
# trainer.py の params に追加
params["class_weight"] = "balanced"
# または
params["is_unbalance"] = True  # LightGBM分類モード用
```

### 6-4. 特徴量の過剰複雑性 🔸中優先

**現状**: 5タイムフレーム × 約100特徴量 = 500+特徴量

LightGBM（`num_leaves=63`）に対して特徴量が過剰な可能性がある。

**改善案**:
1. SHAP の `top_pct` をさらに絞る: `0.5 → 0.3`（上位30%に限定）
2. 相関の高い特徴量を事前除去（SMA/EMAが多重に似た情報）
3. 特定タイムフレームのみに絞るテスト（M5+H1の2TFのみなど）

### 6-5. 時系列分割の改善 🔸中優先

**現状**:
```python
# trainer.py
split_idx = int(len(X) * (1 - val_ratio))  # 0.8/0.2 固定
val_ratio = 0.2  # デフォルト
```

**問題**: Validationが最後の20%固定のため、特定期間に過適合する可能性。

**改善案**: WFO（Walk-Forward Optimization）が既に実装済みなので、
**最終モデル学習前に必ずWFOを実行**して汎化性を確認する習慣を徹底。

`run_wfo_before_train: true` は設定済みだが、
WFO結果（`wfo_min_win_rate: 0.45`）を満たさない場合の対処を明確化する。

### 6-6. 予測ホライズンの検討 🔸中優先

**現状**: `prediction_horizon: 6`（M5×6=30分先の対数収益率を予測）

30分先の方向予測は難易度が高い（マーケットノイズが大きい）。

**検討案**:

| ホライズン | バー数 | 特性 |
|----------|------|------|
| 現行: 30分 | 6 | 高頻度、ノイズ多 |
| 1時間 | 12 | バランス型 |
| 2時間 | 24 | トレンド把握しやすい |

M5で取引するが、ラベルをH1（12バー先）で作るのは有効な改善。

## 7. ポジションサイジングの改善 🔸中優先

### 現状の問題

全37取引が `0.07 lot`（最大値）。

これはロット計算式が常にmax_lotに達していることを意味する：

```python
# position_sizer.py
base_lot = risk_amount / (sl_distance × 100_000)
# 口座残高が大きい、またはATRが小さい場合 → base_lot >> max_lot
```

**結果**: 口座規模に関係なくリスク管理が機能していない。

### 改善案

1. **`max_lot` を口座残高に連動させる**:
   ```yaml
   # 例: 残高の0.5%をmax_lotとして動的計算
   # balance=1,000,000 → max_lot=0.05
   # balance=100,000  → max_lot=0.005
   ```

2. **`max_risk_per_trade` を下げる**: 2% → 1%（損失時の影響を抑制）

3. **シンボル別ロット上限**: EURJPYなど成績不振のシンボルのみmax_lotを下げる

## 8. 市場フィルターの改善 🔹低優先

### ADXフィルター

`min_adx: 25` は標準値だが、以下の課題がある:

- ADX は遅行指標であり、トレンドが始まったばかりでは ADX<25 の場合がある
- レンジ相場でも一時的に ADX>25 を示すことがある

**改善案**:
- ADX の方向性（上昇中か下降中か）も条件に追加:
  `adx > 25 AND adx_delta > 0`（ADXが上昇傾向のとき）

### スプレッドフィルター

`max_spread_pips: 3.0` は現状適切。
EURJPYは通常1.5〜2.5pipsであり、3.0は余裕がある。
EURJPYを絞る場合は `max_spread_pips: 2.0` を検討。

## 10. 総括

**最も重要な発見**:

1. **ラベル不整合（#1）**: 学習時のTP=2.0×ATR vs 取引時のTP=3.0×ATR。
   これは直ちに修正すべきバグレベルの問題。

2. **トレーリングストップの設計ミス（#2）**: 発動後もエントリー価格以下でSLが設定されるため、
   「利益を守る」という本来の目的を果たせていない。

3. **EURUSD は優秀**: 勝率63.6%、+6,045。このモデルの方向性は正しい。
   USDJPY/EURJPYの足を引っ張りを解消することが全体改善の近道。

4. **Regression モードの限界**: confidence=1.0固定で信頼度フィルタが機能していない。
   Classificationモードへの切り替えと`min_confidence`の活用が中期的な改善鍵。
